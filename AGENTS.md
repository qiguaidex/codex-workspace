# Codex CODE 全局约束 - VIBE CODING 增强规则

## 工作区文件（启动时必读）

**每次新会话开始时，必须先读取以下两份文件建立上下文：**
- D:\workspace\WORKSPACE.md — 项目索引、命名规范、交付规则、技能清单
- D:\workspace\PITFALLS.md — 历史踩坑记录、反模式、解决方案沉淀

## 语言规则

**所有对话、文字标注、代码注释、文件内容、交付说明均使用中文。** 仅代码标识符（变量名、函数名、类名等）、技术术语和命令行指令保留原文。

## 核心原则：小步迭代，及时报告

在 vibe coding 场景下，你的首要目标是**确保用户能够验证每个功能点的实现效果**，避免错误堆积导致项目最终无法运行。

### 关键理念
- **用户掌握运行环境，你掌握代码质量**
- **用户是你的最终验证者** — bash 输出往往无法判断程序是否正常运行
- **每轮结束时交付可验证的产物**，而非每步都中断等待

---

## 规则 1：任务拆分的原子化要求

### 拆分粒度标准
将一个任务拆分为**原子功能点**：每个功能点应该是一个"可独立验证的完整行为"。

**判断标准**（非硬性数字）：
- 完成后用户能理解"这个功能点做了什么"
- 如果拆分会导致用户无法看到完整效果，则不拆分
- 如果多个功能点互不依赖、可并行验证，可以合并到同一轮交付

### 拆分判断流程
`
1. 这个任务可以分解为哪些可独立验证的功能点？
2. 第一个功能点是否足够独立，可单独验证？
3. 如果不能 — 再拆一级
`

---

## 规则 2：阶段性交付与用户验证

### 执行节奏
在一轮对话中尽可能完成多个原子功能点，**轮末统一报告**：

1. 完成本轮所有功能点
2. 汇总告知用户完成了哪些功能点
3. 提供每个功能点的具体验证步骤
4. 用户自行验证，发现的问题在下一轮优先处理

### 交付报告模板
`
✅ 本轮完成:
1. [功能点名称] — [验证步骤]
2. [功能点名称] — [验证步骤]

📋 请验证以上功能，有问题随时告诉我。
`

### 验证方式矩阵

| 变更类型 | 验证方式 | 是否需用户参与 |
|---------|---------|--------------|
| UI 变更（HTML/CSS/组件） | 用户目视检查 | 是 |
| 前端交互逻辑（JS） | 用户操作验证 | 是 |
| 后端 API / 数据处理 | 自动化测试或命令输出 | 否 |
| 数据库 / 配置文件 | 命令执行 + 输出检查 | 否 |
| 代码重构 | 测试通过 + 行为不变 | 否 |
| 纯文档 / 注释 | diff 检查 | 否 |

### 关键约束
- **禁止连续完成多个用户可见的功能点后才提供报告** — 但可以在同一轮内完成
- **用户报告问题时，立即停止新功能，优先修复**
- **bash 无报错 ≠ 功能正确**，需配合验证步骤确认

---

## 规则 3：设计文档的处理策略

### 何时需要设计文档
仅当任务满足以下**任一**条件时，主动提供简要设计概要：
- 涉及 3 个以上新文件
- 架构不清晰，有多种可行方案
- 用户明确要求

### 设计概要标准
极简化，只包含：

**必须包含**：
- 核心功能列表（3-5 个关键点）
- 主要文件结构
- 关键技术选型（如果有）

**禁止包含**（除非用户明确要求）：
- 详细测试用例
- 错误处理方案
- 性能优化策略
- 过度详细的 API 设计

---

## 规则 4：用户反馈的优先级

### 反馈处理原则
`
用户的执行反馈 > bash 输出 > 代码逻辑推测
`

- 当用户报告功能有问题时，即使 bash 没有报错，也应立即排查
- 将用户视为最终的"运行环境真相"
- 下一轮对话优先处理上一轮遗留的问题

### 反馈响应流程
1. 接收反馈 → 明确问题
2. 暂停新功能 → 定位修复
3. 修复 → 让用户确认

---

## 规则 5：错误预防机制

### 主动检查点
在以下情况下，确保提供验证步骤：
- 完成 UI 相关的代码（HTML、CSS、前端组件）
- 修改了关键配置文件（package.json、requirements.txt 等）
- 连接了外部服务或 API
- 实现了核心业务逻辑

### 风险识别
如果出现以下情况，主动告知用户并建议拆分：
- 功能点之间存在强依赖关系
- 无法在当前轮次内完成一个完整的功能闭环

---

## 规则 6：问题排查方法

遇到阻塞时，**禁止直接说"做不了"**。必须逐层排查，定位唯一的真正卡点。

### 排查原则
- 每一层失败不能否定所有层 — 网络不通 ≠ API 不通 ≠ 认证不通
- 区分"大文件下载超时"和"小请求超时"，前者可用替代渠道（npm 镜像、curl API）
- 只将用户无法代劳的事交给用户（如输入密码、创建 Token），其余自己解决

### 排查流程

```
遇到阻塞
  → 第1层: 网络通不通？（ping / curl 小请求测试）
     ├─ 通 → 继续
     └─ 不通 → 检查代理/镜像，再试
  → 第2层: 工具装没装？（which gh / npm --version / winget）
     ├─ 有 → 用现有工具
     └─ 没有 → 下载安装，或找替代渠道（npm 代替 winget）
  → 第3层: 小请求通不通？（curl api.github.com / curl registry.npmjs.org）
     ├─ 通 → 大文件下载慢不是根本问题，改用 API 或轻量方案
     └─ 不通 → 网络确实不通，报告用户
  → 第4层: 认证有没有？（credential.helper / 环境变量 / token 文件）
     ├─ 有 → 直接用
     └─ 没有 → 这是真正的卡点，明确告诉用户需要什么
```

### 报告格式
向用户报告阻塞时，必须同时给出：
- 最终卡点是什么（一个点，不是一个清单）
- 排查过的所有层及结果（证明不是其他层的问题）
- 用户需要做什么（最小动作，如"创建一个 GitHub Token，勾选 repo 权限"）

### 事后沉淀
问题解决后，将新发现的卡点类型追加到 `D:\workspace\PITFALLS.md`，防止下次重复排查。

## 文件写入规范

### 编码要求
- **所有文件必须以 UTF-8（无 BOM）编码写入**
- 写入中文内容后，立即检查文件字节是否出现连续的 0x3F（ASCII ?）— 出现则表示编码损毁
- 禁止使用 PowerShell 的 Out-File / Set-Content 默认编码写入含中文的文件，始终显式指定 -Encoding UTF8

### 文件路径
- Windows 路径使用 D:\workspace\...，不要用 D:\Workspace\...（大小写一律小写）

## 特殊说明

### 与 Codex Code 现有规则的关系
- 本规则**不改变** Codex 的自主完成机制（核心原则：在一轮内尽量完成）

- 用户反馈的优先级始终最高

### 适用场景
这些规则主要针对 **vibe coding 场景**：
- 快速原型开发
- 个人项目构建
- 学习和实验性项目
