# 全局踩坑日志

> 与 `AGENTS.md`（VIBE CODING 增强规则）配合使用：AGENTS.md 定义「该怎么做」，本文件记录「做错了什么」。
> Codex AI 在每次新会话时应搜索与当前任务相关的标签，避免重复踩坑。

---

## 使用说明

### 对人类
- 遇到任何错误或异常行为，按下方模板记录
- 即使当时解决了也要记，未来可能复现
- 不确定原因也记录，标注「原因未知」

### 对 AI
- 新会话开始时搜索相关 `#标签`
- 执行任务前检查是否有历史问题
- 遇到新问题，解决后追加到本文件
- 如果发现某个错误模式反复出现，追加到「反模式」部分

---

## 问题模板

```markdown
## YYYY-MM-DD | 项目 | 问题简述

- **标签**: #tag1 #tag2
- **现象**: 实际看到了什么异常
- **原因**: 为什么会发生
- **解决**: 具体做了什么
- **教训**: 如何预防，或者下次怎么更快定位
```

### 2026-06-08 | 工作区 | 文件被进程占用无法删除

- **标签**: `#windows` `#workaround`

### 2026-06-09 | 工作区 | 元文件中文编码批量损坏

- **标签**: `#encoding` `#powershell` `#bug`
- **现象**: `D:\workspace\AGENTS.md`、`WORKSPACE.md` 尾部、`.gitattributes` 注释、`Documents\Codex\AGENTS.md` — 四处中文内容全部变为 `?`（0x3F 字节替换），原始汉字不可恢复
- **原因**: 文件在 PowerShell 中通过 `Set-Content` / `Out-File` 或管道重定向写入时未显式指定 `-Encoding UTF8`，PowerShell 默认使用 ASCII/ANSI 编码，多字节 UTF-8 汉字被替换为 0x3F
- **解决**: 从 `C:\Users\Administrator\.codex\AGENTS.md`（UTF-8 BOM 原版）恢复 AGENTS.md；WORKSPACE.md 尾部从"十一"节已有内容重建；`.gitattributes` 注释手动重写
- **教训**: (1) 所有文件写入必须显式指定 `-Encoding UTF8`；(2) AGENTS.md 设置"文件写入规范"章节防止复发；(3) 写入含中文的文件后立即用 `[System.IO.File]::ReadAllBytes` 检查 0x3F 计数

## Vibe Coding 反模式（重要）

> 以下是在 vibe coding 中实际发生过的错误工作方式。规则定义见 `D:\workspace\AGENTS.md`，此处仅记录实例。

### 反模式 1：连续编写不验证

- **标签**: `#vibe` `#vibe-verify`
- **现象**: AI 连续写多个文件后才让用户验证，结果前面的代码就有问题，后面全部白写
- **后果**: 错误堆积，排查困难，可能需要大量回退
- **正确做法**: 每完成一个原子功能点就暂停，按交付模板要求用户验证
- **触发信号**: 一次修改超过 3 个文件、新增超过 100 行代码

### 反模式 2：过度工程化

- **标签**: `#vibe` `#vibe-overengineer`
- **现象**: 动手前写长篇设计文档，包含测试计划、错误处理、性能优化等
- **后果**: 浪费时间在不会用到的设计上，真正该写的代码还没开始
- **正确做法**: 先问 A/B，选择 B 直接写代码，或选择 A 写极简文档（3-5 功能点 + 文件结构 + 技术选型）
- **触发信号**: 设计文档超过一页、包含测试策略或部署方案章节

### 反模式 3：假设 bash 成功 = 功能正确

- **标签**: `#vibe` `#vibe-verify`
- **现象**: 命令执行无报错，AI 认为功能正常，直接继续下一步
- **后果**: 命令不报错不等于程序按预期运行，用户实际看到的可能完全不同
- **正确做法**: bash 执行后明确告诉用户如何验证实际效果，等待反馈
- **触发信号**: 执行完 `npm run dev` 或 `python app.py` 后没有要求用户在浏览器/终端验证

### 反模式 4：忽略用户反馈继续写新功能

- **标签**: `#vibe` `#vibe-verify`
- **现象**: 用户报告问题，AI 说「这个问题稍后处理」然后继续写新代码
- **后果**: 问题积累，用户失去信心，项目最终无法运行
- **正确做法**: 用户反馈优先级最高（> bash 输出 > 代码推测），立即停新功能，定位修复

## 解决方案沉淀

### 模式：Windows 文件占用处理
```
场景: 文件/目录无法删除或移动，报"被另一程序占用"
步骤:
  1. robocopy <源> <目标> /E /MOVE /R:2 /W:1
  2. 忽略"删除源目录失败"错误
  3. 重启后空壳目录自动消失
```

### 模式：Codex 配置修改
```
场景: 修改 Codex 行为配置
文件: C:\Users\Administrator\.codex\config.toml
要点: 修改前备份 → 修改后重启 → 启动后验证
```

### 模式：新项目初始化清单
```
1. 创建目录（按 Projects-NN-中文说明 格式）
2. git init && git checkout -b main
3. 创建 .gitignore（至少排除 node_modules/ .codex/）
4. 创建 README.md
5. 首次提交
6. 创建 GitHub 仓库并推送
```
