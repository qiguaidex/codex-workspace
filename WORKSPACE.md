## 四、命名规范

### 4.1 项目命名
- **格式**: `Projects-NN-中文说明`（NN 为两位数字序号）
- **正面示例**: `Projects-01-个人助理`、`Projects-05-数据看板`
- **禁止**: 英文名、拼音、含空格、大小写混用

### 4.2 技能命名
- **自定义技能格式**: `阿木-功能名称`
- **正面示例**: `阿木-视频制作`、`阿木-飞书助手`
- **禁止**: 与已有技能重名、使用英文名

### 4.3 文件命名
- 脚本：英文小写 + 下划线（`send_papers.py`、`feishu_api.js`）
- 文档：中文或英文，不含空格（`设计概要.md`、`README.md`）
- 配置：遵循工具默认（`package.json`、`.gitignore`）

### 4.4 Git 分支
- 主分支统一使用 `main`
- 功能分支：`feat-简短描述`
- 修复分支：`fix-简短描述`

## 五、Git 规范

### 5.1 铁律
1. 每个功能点完成后立即提交，禁止堆积
2. 提交前确认 `git status` 干净
3. 禁止提交 
ode_modules/`、`.codex/`、日志、临时文件
4. 每个项目必须有 `.gitignore`

### 5.2 提交格式
```
<类型>: <中文简述>

feat     — 新功能
fix      — 修复 bug
refactor — 重构（不改变功能）
docs     — 文档
chore    — 杂项
```

### 5.3 远程仓库
- 每个项目应在 GitHub 有对应远程仓库
- 仓库名与项目目录名一致
- 推送前确认无敏感信息

## 六、VIBE CODING 交付规则（核心）

> 详细规则见 `D:\workspace\AGENTS.md`。本节仅列出工作区级强制约束。

### 6.1 原子化拆分

在执行任何任务前，必须拆分为**原子功能点**：
- 一个功能点 = 用户可以立即验证的最小单元
- 完成后用户能实际运行并看到变化
- 禁止：一个功能点修改 5 个以上文件或超过 100 行新代码

```
❌ "实现用户登录功能"
✅ 拆分:
   → 创建登录表单 HTML 结构
   → 添加表单验证逻辑
   → 连接后端 API 并处理响应
```

### 6.2 新任务启动流程

收到新任务时，**先确认**：

> 您希望我：A. 先创建简要设计文档作为指导  B. 直接开始编写代码

文档必须极简：核心功能 3-5 点、主要文件、技术选型。禁止过度工程化。

### 6.3 阶段性交付

每完成一个原子功能点后，必须按以下格式暂停：

```
✅ 已完成: [功能点名称]

📋 请验证:
1. [具体验证步骤]
2. ...

🎯 预期效果:
[用户应该看到什么]

⏸️ 请确认此功能是否正常，然后我会继续下一个功能点。
```

**强制约束**:
- 禁止连续完成多个功能点后才要求验证
- 禁止假设功能正确而直接继续
- 即使 bash 无报错也必须让用户验证

### 6.4 用户反馈优先级

```
用户的执行反馈 > bash 输出 > 代码逻辑推测
```

用户报告问题时，即使 bash 没报错也立即排查。不要在新功能堆积时忽略问题。

### 6.5 写前写后检查

**写代码前自问**:
- 是否询问了设计文档（A/B）？
- 任务是否已拆分为原子功能点？
- 用户如何验证这个功能点？

**写代码后自问**:
- 是否已告知用户完成了什么？
- 是否提供了具体验证步骤？
- 是否在等用户反馈而非自行继续？

## 七、Codex 行为契约

### 7.1 代码位置
- 所有代码写入 `D:\workspace\Projects-NN-xxx/`
- 临时脚本放项目内的 `work/` 或 `scripts/`
- 交付文件放项目内的 `outputs/`
- 禁止在根目录创建代码文件

### 7.2 语言
- 对话、注释、文档全部中文
- 代码标识符保留英文

### 7.3 错误处理
- 遇到问题先查 `PITFALLS.md` 是否有记录
- 解决问题后追加到 `PITFALLS.md`
- 不确定立即问用户，不要猜测后继续
